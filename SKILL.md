## 🎯 核心目标

**一句话**: 决策者（示例中称为"王爷"，实际使用时请替换为你的角色名称）创建日程 → 妙记自动生成 → **全自动**逐字稿 + 会议纪要 + 任务提取 + 多维表格归档 + 逾期追踪 + 会前关联。

**解决痛点**:
- 开完会后没人系统性记录：谁该做什么？什么时候做完？
- 会议纪要散落在各处，下次开会找不到
- 任务说了就忘了，没人追踪执行情况
- 同类会议反复开，上次定的事没人记得

**核心价值**:
1. **自动归档**: 会议结束 → 妙记 → 逐字稿 + 6 份文档 → 多维表格，全程无需手动
2. **任务闭环**: 提取 P1 任务 → 录入任务表 → Cron Job 逾期提醒
3. **跨会议关联**: 下次开会前 24h 自动推送"上次定了什么 + 哪些没完成"
4. **智能分发**: 决策给高管、任务给执行人、知悉给相关人等（所有信息抄送上级，外部人员不发送）

## 🧠 LLM 价值提取规则

### 1. 核心原则：双重判断法

**视角一：后果驱动**（用于定级 P0/P1/P2）
- **不做会怎样？** 从"确定性"出发——已经产生了决策、部署了任务、分享了明确信息
- 后果越严重，等级越高

**视角二：信号捕捉**（用于定级 P0.5）
- **是否值得关注的信号？** 从"不确定性"出发——CEO 的担忧、两人的分歧、模糊的外部变化
- 特征是"没有明确后果，但有危险的感觉"

> 💡 **80% 的会议价值藏在信号捕捉中**

---

### 2. 六级价值量定义

| 等级 | 标签 | 含义 | 判断依据 | 触发动作 |
|:---:|:---:|:---|:---|:---|
| **P0** | 🔴 战略级 | 涉及公司方向、重大决策、CEO 拍板 | 不做公司会出大事 | → 《战略决策备忘》 |
| **P0.5** | 🔶 风险级 | 会上提了但没解决的风险、分歧、担忧 | 不确定后果，但感觉到了危险 | → 《风险与问题台账》 |
| **P1** | 🟠 行动级 | 需要具体某人限时完成的明确任务 | 不做会亏钱、丢机会、耽误进度 | → 《个人执行任务单》 |
| **P2** | 🟡 知悉级 | 不需要行动，但与某些人相关 | 知道了有好处，不知道也不耽误 | → 《信息分发快报》 |
| **P3** | 🟢 背景级 | 背景信息、行业趋势、参考观点 | 有价值但不紧急 | → 《信息分发快报》附录 |
| **P4** | ⬜ 无效级 | 闲聊、寒暄、重复讨论 | 完全无关 | → 直接丢弃 |

**分类逻辑**:
```plaintext
P0/P0.5 → 决策者参考（高管层，示例中称为"王爷"）
P1 → 执行人行动指南（执行层）
P2/P3 → 相关人员知悉（知悉层）
P4 → 丢弃（无价值）
```

---

### 3. P0.5 风险级识别规则

**出现以下任一即标记 P0.5**:
- ✅ 出现"这个问题很大，但我们还没想好"、"先放一放"、"这个要关注"
- ✅ 两人以上对同一问题立场相反，未达成共识
- ✅ CEO 表达了担忧但未形成行动指令
- ✅ 外部变化（政策/竞品/供应链）被提及但无应对方案
- ✅ 会上有人反复提起某个问题，但始终没有展开讨论

**AI 处理方式**:
- AI 识别这些信号后标记为 P0.5 候选，附上原文引用
- 决策者（示例中称为"王爷"）在审核时确认：这条真的需要进台账追踪？还是降为 P2 知悉即可？

---

### 4. 六维度判断参考

| 维度 | 考察问题 | 主要用于 |
|:---|:---|:---|
| **决策相关性** | 是否产生了明确决策？ | 区分 P0/P0.5 |
| **行动可执行性** | 能拆成"谁 + 做什么 + 何时"？ | 区分 P1/P2 |
| **人员关联度** | 与哪些人的工作直接相关？ | 决定分发给谁 |
| **时效紧迫性** | 本周内必须处理？ | 区分 P0.5/P1 vs P2/P3 |
| **信息稀缺性** | 这是否只在这个会议才知道？ | 提高等级 |
| **可验证性** | 有明确验收标准？ | 区分 P1/P2 |

---

### 5. 任务六要素（P1 强制，缺一不可）
```plaintext
▎任务：[具体动作——能干什么，不是方向性描述]
▎协同：[和谁配合 / 依赖谁]
▎产出：[验收标准——什么叫做完了？什么叫做好了？]
▎截止：[精确到日期，不允许"尽快"、"下周"]
▎反馈：[向谁汇报 / 汇报形式 / 中途反馈时间点]
▎为何重要：[不做会有什么后果]
```

**示例对比**:
❌ **坏**："小张，把那个系统对接一下。"
✅ **好**:
- **任务**：完成运营系统与财务结算系统的数据接口对接
- **协同**：技术部配合，需数据库只读权限
- **产出**：订单→收款→退款三条链路打通，测试环境 100 笔全流程无报错
- **截止**：2026-05-15 18:00
- **反馈**：5 月 13 日 18:00 飞书向项目负责人汇报联调进展
- **为何重要**：不做则本月财务对账继续手工 Excel，已连续三个月出错

### 6. AI 处理边界与兜底

| 情况 | 处理方式 |
|:---|:---|
| 录音提到"那个事"，无法推断 | 标注"待补充：请确认具体事项" |
| 未提截止时间 | 标注"截止时间：待确认" |
| 名字不明确（"小王"） | 标注"小王（待确认全名）" |
| 无法判断发给谁 | 标注"建议知悉：待确认" |

**核心原则**：宁可标"待确认"，绝不编造。

### 7. 降级模式说明 ⚠️

**触发条件**: 当 LLM API 不可用时，自动降级到关键词匹配模式（`extractValueSimple()` 函数）。

**局限性**:
- 关键词匹配无法理解语义，只能基于表面词汇判断
- 专业讨论场景下 P0.5/P1 提取准确率显著下降（约 40-60%）
- 无法识别复杂的决策逻辑和任务分配关系
- 无法提取六要素（任务/协同/产出/截止/反馈/为何重要）

**建议**:
- ✅ 确保 LLM API 可用以获得最佳效果（准确率 85%+）
- ⚠️ 降级模式仅作为应急兜底，不建议长期依赖
- 📊 降级模式下提取的结果需要人工复核

## 🔍 会前检测机制

### 1. 会前三重检测

| 检测 | 逻辑 | 输出示例 |
|:---|:---|:---|
| **重复检测** | 议题与近 3 次会议重合 | "此议题在 4 月 29 日已有决策：XXX。本次是更新还是推翻？" |
| **矛盾检测** | 本次方向与历史相反 | "与 4 月 29 日决策方向相反。是新信息改变了判断，还是策略调整？" |
| **遗漏检测** | 上次 P1 负责人出席了但未提进展 | "上次 P1 任务'XXX'（负责人：XX，截止：X 月 X 日）未被追踪。建议确认执行状态。" |

### 2. 会前匹配规则
```plaintext
标题相似度 ≥ 60%（提取 2 字以上中文词）
或 参会人重合度 ≥ 50%
或 同一日程系列（飞书重复日程）

兜底机制:
  自动匹配失败 → 提醒用户："发现新日程 [标题]，是否需要生成会前文档 E/F？"
  用户回复"生成" → 手动触发
```

### 3. 冷启动机制
- **第 1 次使用**: 标注"首次使用，跨会议关联将在下次会议后启用"
- **第 2 次**: 展示简单对比
- **第 3 次后**: 三重检测全开

### Step 6.5: 权限自动设置
**命令**:
```bash
lark-cli drive permission.public patch \
  --token [DOC_TOKEN] \
  --type docx \
  --data '{"link_share_entity":"tenant_readable"}' \
  --yes
```

**实现代码** (index.js):
```javascript
async function setDocumentsPermission(docTokens) {
  for (const token of docTokens) {
    const cmd = `lark-cli drive permission.public patch --token ${token} --type docx --data '{"link_share_entity":"tenant_readable"}' --yes`;
    await exec(cmd);
  }
}
```

### Step 9.5: 任务录入多维表格
**目标表格**: 会议任务管理表（需在配置中设置）

#### 1. 录入命令
```bash

    "来源会议": "xxx",

  }'
```

> ⚠️ **注意**: 
> 1. 请将 `[你的任务管理表 baseToken]` 和 `[你的任务管理表 tableId]` 替换为实际值
> 2. 建议在 `config.json` 中配置 `tables.task_management_base_token` 和 `tables.task_management_table_id`
> 3. 命令格式：`lark-cli base record.create --base-token XXX --table-id YYY --fields 'JSON'`

#### 2. 自动化规则（CLI 创建）✅
**触发条件**:
```plaintext
截止时间 = 今天 + 3 天（即到期前 3 天提醒）
AND
当前状态 ≠ 已完成
```

**执行动作**:
```plaintext
发送飞书消息 → 负责人
```

**CLI 命令**（创建自动化工作流）:
```bash
lark-cli base +workflow-create \
  --base-token [请替换为你的任务管理表 baseToken] \
  --title "任务到期前 3 天提醒" \
  --client-token "task_reminder_3days" \
  --steps '[
    {
      "type": "ChangeRecordTrigger",
      "params": {
        "tableId": "[请替换为你的 tableId]",
        "conditions": [
          {
            "fieldId": "截止时间",
            "operator": "equals",
            "value": "today_plus_3"
          },
          {
            "fieldId": "当前状态",
```

```javascript
async function recordTasksToBase(minuteToken, meetingTitle, meetingDate, p1Tasks, config) {
  // 配置优先级：config.json > 环境变量 > 报错
  const baseToken = config.tables?.task_management_base_token || config.baseToken || process.env.MEETING_TASK_BASE_TOKEN;
  const tableId = config.tables?.task_management_table_id || config.taskTableId;
  
  if (!baseToken) {

  }
  
  if (!tableId) {

    return;
  }
  
  for (const task of p1Tasks) {
    const fields = {
      '任务内容': task.content,
      '负责人': task.owner || '待确认',
      '截止时间': task.deadline || '待确认',
      '来源会议': meetingTitle,
      '妙记 token': minuteToken,
      '创建日期': meetingDate,
      '当前状态': '待开始'
    };
    const cmd = `lark-cli base record.create --base-token ${baseToken} --table-id ${tableId} --fields '${JSON.stringify(fields)}'`;
    await exec(cmd);
  }
}
```
